---
name: Kontra Workflow Pack architecture
description: How Kontra deal-room behavior (roles, stages, checklist, health scoring, invites, dashboard visibility) is driven by a pluggable "Workflow Pack" rather than hardcoded per-component, and validated with two packs (CRE + Business Acquisition).
---

Kontra's deal room used to hardcode CRE-specific roles/stages/checklist/health logic separately inside `DocumentChecklistPanel.jsx`, `DealHealthPanel.jsx`, `DealCoordinationPanel.jsx`, and `InvitePanel.jsx`. That coupling is extracted into `kontra-ui-clone/ui/src/lib/workflowPacks/` — each pack module (`creAcquisition.js`, `businessAcquisition.js`) defines roles/stages/document schema/health scoring, and `index.js` exposes `getWorkflowPack(packId)` / `DEFAULT_PACK_ID` / `listWorkflowPacks()`. (Previously named `workflowTemplates/` / `getWorkflowTemplate` — renamed to "Workflow Pack" terminology; the dir was fully replaced, not kept alongside.)

**Why:** the four panels each had their own near-duplicate role list, so any new workflow (servicing, refinance, hazard loss, or an entirely different domain like Business Acquisition) would have required editing 4+ files in sync. Centralizing lets new packs be added by writing one pack module and registering it in `index.js` — the panels don't change. Business Acquisition was added as a second pack specifically to validate the engine is domain-agnostic, not just CRE-flavored.

**How to apply:**
- Any new UI that needs role/stage/checklist/health data should pull it from `getWorkflowPack(packId)` (default `DEFAULT_PACK_ID`), not redefine its own list.
- `packId` flows: chosen once at deal-room creation (`CreateDealRoomPage.jsx`) → sent as `meta.workflowPackId` in the checkout payload → persisted on `deal_rooms.workflow_pack_id` (migration `005_workflow_pack_id.sql`, defaults `'cre_acquisition'`) → read back via `DealRoomPage.jsx`'s `packId = demoProperty ? DEFAULT_PACK_ID : (apiProperty?.workflow_pack_id || DEFAULT_PACK_ID)` and passed as a prop into the panels.
- `DealIntelligenceDashboard` and `FinancialSnapshotPanel` are now pack-driven (Sprint 3): each pack exports `intelligenceSections`/`getIntelligenceBadge`/`getIntelligenceHighlight`/`getSnapshotStats`/`getSnapshotFlag`, and both widgets take a `packId` prop and self-render nothing when a pack's section list/stats are empty (Business Acquisition currently defines no-op stubs, so those widgets are invisible for that pack without any gating flag). Any new pack must add these five exports (even as no-ops) or the widgets will crash calling `undefined`.
- The risk/compliance/property "Outstanding Items" grid is still CRE-only and hardcodes NOI/DSCR/occupancy concepts directly in `SECTION_MAP` panels — it remains gated behind `showCreDashboard = packId === DEFAULT_PACK_ID` in `DealRoomPage.jsx`. Making this grid pack-driven too is unstarted.
- Lifecycle stage *keys* (`uploading`, `under_review`, `approved`, `closing`, `funded`) are shared across all packs today because the backend coordination API (`/advance`, `/submissions/:role/status` in `kontra-ui-clone/api/index.js`) validates against one fixed `VALID` stage-key array — packs only vary stage *labels/icons/desc*, not keys. If a future pack needs a genuinely different stage shape, the backend's valid-stage list needs to become pack-driven too.
- Role objects support a `canManage: true` flag (e.g. CRE's `owner`/`lender`, Business Acquisition's `buyer`) read via `ROLE_META[role]?.canManage` in `DealCoordinationPanel.jsx` — this replaced a hardcoded `role === 'owner' || role === 'lender'` check that would've been wrong for non-CRE packs.
- Peripheral pages (`MyDealRoomsPage.jsx`, `CheckoutSuccessPage.jsx`, `DealSharePage.jsx`, `DealSummaryPage.jsx`) still have their own small local role-label maps for cosmetic badges — intentionally out of scope so far, and will show wrong labels for non-CRE packs until migrated.
- Backend `link_token` column was found missing during this work (pre-existing, unrelated gap, silently swallowed by a `.catch`) — fixed via migration `006_link_token.sql`. Both `kontra-ui-clone/api/index.js` (dev) and `api/index.js` (prod/Render) must stay in sync — see kontra-deploy-routing.md.
