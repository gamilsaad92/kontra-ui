Here’s a **developer instruction pack** you can give directly to your engineer for implementing **OpenClaw inside Kontra** as the AI orchestration layer for servicing workflows. Public OpenClaw materials describe it as an open-source personal AI assistant/runtime with tools such as browser, canvas, nodes, cron, sessions, and Slack/Discord actions, plus a skills-based setup model. That makes it a good fit as the orchestration/runtime layer, while Kontra remains the system of record and UI. ([GitHub][1])

## Developer brief

**Mission:**
Implement OpenClaw as Kontra’s **agent runtime and workflow orchestrator** so the platform can automate loan servicing tasks including financial review, inspection review, draw validation, borrower communication, and risk scoring.

**Target outcome:**
When a user uploads a financial package, inspection report, draw request, or borrower document in Kontra, the platform should automatically trigger the correct AI workflow, produce a structured decision, save the result to the database, and surface it in the Kontra UI for human review or approval.

---

## 1) High-level architecture

**Keep responsibilities separate:**

* **Kontra UI** = dashboard, task inbox, review screens, audit trail
* **Kontra API** = source of truth, business rules, auth, DB writes, permissions
* **OpenClaw runtime** = agent execution, tool calling, workflow coordination, scheduled jobs
* **Supabase/Postgres** = workflow state, artifacts, decisions, logs, agent memory summaries

**Rule:**
OpenClaw should **not** become the system of record. It should execute work and return structured outputs back to Kontra API, which validates and persists them.

OpenClaw’s ecosystem appears oriented around tools, sessions, cron, and skills, and there is active community work around multi-agent orchestration and workflow plugins. Use it as the execution fabric, not the database or dashboard. ([GitHub][1])

---

## 2) What to build first

Implement these five agents as separate modules:

1. **Financial Agent**

   * Reads financial statements, rent rolls, trailing statements
   * Calculates DSCR and key variances
   * Drafts a watchlist comment
   * Flags missing data

2. **Inspection Agent**

   * Reads inspection reports and image evidence
   * Classifies issues into categories like life safety, deferred maintenance, PR-90, monitoring item
   * Drafts recommended follow-up language

3. **Draw Agent**

   * Validates invoices, lien waivers, proof of payment, repair photos, contractor support
   * Detects missing or inconsistent documents
   * Recommends approve / conditional approve / reject

4. **Borrower Communication Agent**

   * Drafts deficiency notices, RORs, follow-up emails, and status updates
   * Pulls from structured findings generated by the other agents

5. **Risk Agent**

   * Aggregates data from financial, inspection, draw, and delinquency events
   * Produces a risk score and reason codes
   * Suggests watchlist or special-servicing escalation when thresholds are breached

## 3) Required implementation pattern

Tell the developer to use this exact pattern for every workflow:

### A. Event comes into Kontra

Examples:

* financial package uploaded
* inspection report uploaded
* draw request submitted
* occupancy drops below threshold
* missing annuals aged more than X days

### B. Kontra API creates workflow record

Create a row in `workflow_runs` with:

* `workflow_type`
* `loan_id`
* `source_entity_type`
* `source_entity_id`
* `status = queued`
* `requested_by`
* `input_payload`

### C. OpenClaw picks up the job

The OpenClaw worker/runtime receives:

* workflow type
* minimal structured context
* references to files/data
* instructions for required output schema

### D. Agent executes with tools

Agent can:

* fetch structured loan data from Kontra API
* read files from storage
* write interim notes
* call subordinate agents if needed
* request retry or human review

### E. Kontra API validates result

The API must reject malformed output, enforce schema, and apply business rules before saving results.

### F. UI displays reviewable result

Every output should appear in a task/review queue with:

* summary
* confidence
* reason codes
* source evidence
* action buttons
* audit history

## 7) Orchestration rules

Tell the developer to implement orchestration in **layers**, not a giant monolith.

### Layer 1: trigger router

Maps event type to workflow type.

### Layer 2: workflow planner

Determines which agent or sequence of agents should run.

Examples:

* financial upload → financial agent → risk agent
* inspection upload → inspection agent → borrower communication agent
* draw submitted → draw agent → borrower communication agent
* repeated weak performance → financial agent → risk agent → escalation recommendation

### Layer 3: verifier

Before marking complete:

* validate JSON schema
* verify required evidence exists
* confirm mandatory fields
* assign confidence score
* decide whether to route to human review

There is active community discussion around OpenClaw multi-agent orchestration, pipeline patterns, and workflow plugins. That supports using a planner + handoff + verification model rather than one huge prompt. ([GitHub][2])

## 8) Skills and tool wrappers to expose to OpenClaw

Since OpenClaw is skills/tool oriented, define Kontra-specific tools for it. Public OpenClaw materials emphasize skills plus built-in tooling/runtime concepts, so this is the cleanest integration style. ([GitHub][1])

Build these tool wrappers:

* `get_loan_summary(loan_id)`
* `get_loan_documents(loan_id, filters)`
* `read_document(file_id)`
* `save_artifact(workflow_run_id, artifact_type, content)`
* `create_review_task(workflow_run_id, reviewer_role)`
* `send_borrower_draft(workflow_run_id, draft_type)`
* `calculate_dscr(financial_payload)`
* `compare_to_underwriting(loan_id, actuals)`
* `get_property_timeline(loan_id)`
* `log_agent_step(workflow_run_id, step_name, payload)`
* `upsert_agent_memory(loan_id, memory_payload)`

The developer should keep these tools narrow, deterministic, and permission-checked.

## 10) Guardrails the developer must implement

This part is critical.

### Never let the agent:

* send emails directly without Kontra approval
* approve fund disbursement without rule validation
* overwrite source records without audit trail
* fabricate missing values
* finalize conclusions without source evidence

### Always require:

* source citation references within the workflow evidence model
* confidence scoring
* human review for medium/high impact actions
* immutable logs for every agent step
* retries with backoff for transient failures

### Confidence thresholds

* `>= 0.92` and low-risk → can auto-draft
* `0.75–0.91` → human review required
* `< 0.75` → incomplete / escalate

## 12) Developer acceptance criteria

Tell the developer the implementation is only “done” when all of this is true:

1. Uploading a financial package automatically creates a workflow run.
2. The financial agent returns valid structured JSON.
3. The result is saved in Kontra and visible in the servicing UI.
4. A draft comment is generated and reviewable.
5. Missing items and evidence are shown.
6. Reviewer can approve, edit, or reject the output.
7. Every step is logged.
8. Failed workflows can be retried safely.
9. No direct side effects happen without approval.
10. The system works per loan and at portfolio scale.
