Implement the Kontra mobile app as a focused mobile product, not a compressed version of the full desktop platform. For the initial release, only build the Investor portal and the Borrower portal as first-class mobile experiences. Do not mirror the Lender and Servicer desktop interfaces into mobile at this stage. The mobile app should be designed around fast actions, status visibility, notifications, uploads, and approvals that make sense on a phone. The desktop web platform should remain the primary environment for complex servicing operations, portfolio analytics, policy administration, exception management, and bulk workflow review.

For the Borrower mobile experience, build a simple workflow-centered app that allows users to submit draw requests, upload documents, upload site photos, view request status, respond to conditions, review notices, and message the servicing team. Prioritize camera integration, file upload, mobile-friendly forms, push notifications, and a clean status tracker for each request. The borrower should be able to open the app and immediately understand what is pending, what was approved, what needs correction, and what action is required next. Include a document center, request history, communication thread, and alerts for missing items, approvals, rejections, and deadlines.

For the Investor mobile experience, build a portfolio monitoring and engagement app. Investors should be able to view token holdings or investment positions, distributions, transaction history, portfolio summaries, governance items, voting actions, and market or asset alerts. The investor app should emphasize visibility and responsiveness rather than deep configuration. Add push notifications for distributions, governance votes, asset updates, covenant events, and relevant loan milestones. Keep the mobile layouts simple, with strong dashboard cards, asset detail pages, transaction logs, and vote confirmation flows.

For the Lender and Servicer roles, do not build full mobile portals yet. Instead, if mobile support is needed, create only limited action modules later such as approval notifications, urgent alerts, simple review actions, and exception acknowledgments. Do not port complex dashboards, large queues, policy editing, multi-record review tables, or command-center layouts into the first mobile release. These workflows should remain web-first because they require screen space, dense information, and operational depth that phones do not handle well.

Architecturally, the mobile app should use the same backend entities, permissions, and APIs as the web platform, but expose only a reduced mobile scope. Reuse the same authentication system and role model so users land in the correct experience based on their assigned role. If a user is both an investor and borrower, support a role switcher inside the mobile app. If a user is only a lender or servicer, either block unsupported mobile access with a clear message or show a limited alerts-only experience depending on product direction. The backend should enforce role-based access exactly the same way across mobile and web.

From a product structure standpoint, create the mobile navigation around role-specific workflows instead of mirroring desktop menus. For Borrower, recommended tabs are Home, Requests, Documents, Messages, and Profile. For Investor, recommended tabs are Home, Holdings, Activity, Governance, and Profile. The Home screen for each role should act as a priority dashboard showing what needs attention now, recent activity, and upcoming actions. Every screen should be optimized for fast scanning, thumb-friendly controls, and minimal typing.

For notifications, implement a strong push notification system because this is one of the main reasons mobile adds value. Borrowers should get alerts for request status changes, missing documentation, approval updates, inspections, and messages. Investors should get alerts for distributions, governance events, transaction confirmations, portfolio changes, and asset-level updates. Notification tapping should deep-link the user directly into the relevant record or action screen.

For media and field functionality, make borrower workflows mobile-native. Add camera capture for site progress photos, document scanning, upload progress indicators, retry handling, and offline-friendly draft saving where possible. Mobile should feel especially useful for on-site construction and draw workflows, where a borrower or sponsor may need to upload information directly from the property. This is one of the strongest reasons to prioritize the borrower portal on mobile.

For design and UX, do not attempt to duplicate the visual density of the desktop platform. Build a simplified design system for mobile with card-based layouts, clear status chips, short action lists, and concise record summaries. Use progressive disclosure so users first see the summary and only expand into detail when needed. Avoid large tables, deep nested forms, and complicated filter systems in the mobile app unless absolutely necessary.

For implementation order, first define the shared mobile API scope for investor and borrower actions only. Second, build authentication and role routing. Third, build borrower request submission and upload flows. Fourth, build investor holdings, distributions, and governance views. Fifth, integrate push notifications and deep linking. Sixth, polish document handling, message threads, and activity history. Seventh, evaluate whether lightweight lender or servicer approval actions are needed as a later extension. The success metric for mobile should be completion of key actions, notification engagement, and speed of response, not feature parity with desktop.

The main product principle should be this: web is the operations system, mobile is the action and visibility layer. Borrowers and investors should have full mobile experiences because their workflows fit the device. Lenders and servicers should remain web-first until there is a clear case for limited mobile actions. That approach will keep the app cleaner, faster to launch, and more useful in real-world use.
