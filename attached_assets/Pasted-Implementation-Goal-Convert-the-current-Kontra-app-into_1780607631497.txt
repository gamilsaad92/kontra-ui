Implementation Goal:

Convert the current Kontra app into one unified public marketplace + AI tools platform.

Do not build a separate new app. Refactor the existing app so the public website, login, signup, dashboard, marketplace, and old modules all connect into one experience.

Current problem:

Public landing page says Kontra is a CRE AI tools marketplace, but after login users are redirected into the old lender/servicing platform. This creates two disconnected products.

Target flow:

Public Website
→ Sign Up / Login
→ Onboarding
→ Unified Kontra Workspace
→ User sees tools based on account type

Required implementation steps:

1. Update Routing

Create one post-login entry point:

/workspace

After login or signup, all users should be redirected to:

/workspace

Do not redirect users directly to the old dashboard.

Routes should be organized like this:

/                  Public homepage
/ai-tools          Public AI tools page
/marketplace       Public marketplace page
/pricing           Public pricing page
/sign-in           Login
/sign-up           Signup
/onboarding        First-time user setup
/workspace         Main authenticated dashboard
/workspace/tools
/workspace/marketplace
/workspace/properties
/workspace/projects
/workspace/documents
/workspace/reports
/workspace/settings

Legacy lender routes should be moved under:

/workspace/enterprise

or hidden unless the user is an enterprise user.

2. Add Onboarding After Signup

After new user signup, send them to onboarding before the dashboard.

Onboarding questions:

* What best describes you?

  * Property Owner
  * Investor
  * Vendor / Service Provider
  * Consultant
  * Lender / Enterprise
  * Other

* What do you want to do first?

  * Analyze a property
  * Review inspection photos
  * Upload financials
  * Track a damage/claim issue
  * Find vendors
  * List my services
  * Manage a portfolio

Save these fields to the user profile table.

Suggested database fields:

user_type
primary_goal
onboarding_completed
subscription_tier

9. Keep Existing Backend Logic

Reuse existing APIs wherever possible.

Map old data to new labels in the frontend first.

Example:

loan_id can still exist in backend, but frontend label should be Property / Asset.

borrower_id can still exist in backend, but frontend label should be Client / Counterparty.

draw_request can still exist in backend, but frontend label should be Project Funding Review.

Do not break the current data model unless required.

13. Important Compliance Positioning

The app should not present itself as a replacement for a lender, servicer, bank, or asset manager.

Avoid wording like:

* We service loans
* We manage lender portfolios
* We replace servicers
* Freddie Mac servicing automation

Use wording like:

* AI tools for CRE professionals
* Property analysis and workflow automation
* Marketplace for CRE services
* Portfolio insights
* Document and inspection automation
