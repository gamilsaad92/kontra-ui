## Developer instructions for applying the recommended Kontra structure

Refactor Kontra from a feature-heavy product into a clear infrastructure platform organized around four core layers: Core Servicing Operations, AI Automation, Rules and Integrations, and Tokenization. Start by restructuring the information architecture and navigation so the product no longer feels like separate disconnected modules or command centers. The main navigation should be Dashboard, Loans, Servicing, AI Copilot, Compliance Rules, Integrations, Tokenization, Marketplace, Reports, and Admin Center. Merge overlapping pages and remove anything that feels repetitive, especially where dashboards and command centers duplicate the same workflows. Rebuild the product around role-specific portals so each user sees only what is relevant to them: Lender for portfolio oversight and policy control, Servicer for payments, inspections, draws, and escrow, Investor for holdings, distributions, and governance, and Borrower for requests, uploads, and communication. All portals should share the same design system and backend entities, but present different layouts, permissions, and default views.

At the data model level, standardize the platform around core entities including organizations, users, roles, loans, assets, borrowers, draw requests, inspections, payment events, escrow accounts, covenants, documents, communications, policies, AI agent decisions, tokens, distributions, and marketplace transactions. Every workflow should run through a central workflow engine with statuses, timestamps, SLA tracking, ownership, comments, audit history, and exception states. This engine should power draws, inspections, payment review, covenant testing, borrower requests, and hazard or loss recovery so the system behaves consistently across all modules. Build a unified activity log tied to every major record so users can see who did what, when, what AI recommended, what policy triggered, and what final decision was made.

For the AI layer, stop presenting individual agents as separate products and instead group them into one user-facing system called Kontra AI Copilot. Internally, keep the agents modular, such as draw review, inspection review, escrow monitoring, borrower financial analysis, payment anomaly detection, risk scoring, report generation, and borrower communications. Each AI action should produce a structured output with recommendation, confidence, explanation, source inputs, model used, and whether human approval is required. Add a human-in-the-loop review queue so AI never silently changes critical loan outcomes. Developers should create a common AI decision schema and reuse it across all AI-assisted workflows so the experience feels unified instead of pieced together.

For the rules and enterprise layer, build a policy engine that allows admins to create and update operational rules without code changes. Rules should support triggers, thresholds, conditions, escalations, required documents, approval requirements, and notifications. This is where your Freddie Mac, Fannie Mae, servicing, inspection, hazard loss, reserve, and tokenization compliance logic should live over time. Build the rule engine so policy updates can be made by admins from the Admin Center and versioned with effective dates, author, change history, and impacted workflows. The frontend should present these as business rules, not code logic. At the same time, implement an integrations layer with adapters for legacy systems and third-party tools. The adapter architecture should normalize outside data into Kontra’s internal data model so the rest of the system always works from a clean shared schema.

For tokenization, keep it as a separate layer that sits on top of validated servicing data, not as a competing primary workflow. Developers should design tokenization so a loan or pool cannot be tokenized unless required servicing, compliance, data completeness, and audit conditions are met. Build a tokenization readiness score and checklist that verifies collateral data, servicing history, compliance documents, borrower data, cash flow data, and investor controls before issuance is enabled. The investor-facing experience should focus on holdings, distributions, transfer restrictions, transaction history, and governance, while the lender-facing experience should retain operational and approval control. Tokenization should feel like an extension of clean loan infrastructure, not a detached crypto feature.

On the UI side, replace six standalone command centers with one Unified Command Center that has tabs or modules for Operations, Inspection Intelligence, Hazard and Loss Recovery, Compliance, Tokenization, and Admin Policy. This should be one system with shared filters, queues, KPIs, alerts, and approval panels rather than six separate apps. Remove any decorative complexity that does not improve execution. Every page should answer one of three questions quickly: what needs attention, what changed, and what action can I take now.

For authentication and authorization, keep login simple and role-based from the start. Users should choose or be routed into the correct portal after login based on their assigned role and tenant membership. Do not bury the borrower, lender, investor, or servicer portals inside one master dashboard. Permissions should be enforced both in the frontend and backend so users only see data and actions relevant to their role and organization. Create a permissions matrix for view, create, approve, reject, upload, comment, export, configure policy, and manage users across each portal.

For reporting and product story alignment, make sure the dashboard and reports reflect the simplified business narrative. Dashboard should highlight loan operations, AI activity, exception queues, policy breaches, and readiness for tokenization. Reports should be generated from structured backend data and include downloadable audit trails, servicing summaries, compliance status, investor reporting, and workflow analytics. AI cost governance should not be a top-level customer-facing product area; move it into Admin Center as an internal analytics and monitoring module.

Finally, tell the developer to do this in order: first clean up the navigation and page structure, second unify the data model and workflow engine, third consolidate AI into one Copilot framework, fourth build the policy engine and admin regulation layer, fifth standardize portals and permissions, sixth make the Unified Command Center the operational control surface, and seventh position tokenization as the final layer enabled by trusted servicing data. The product should feel like one infrastructure platform from end to end, not a bundle of unrelated tools.
